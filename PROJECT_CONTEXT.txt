1. PROJECT OVERVIEW
================================================================================

Blood Bank Pro is a full-stack web-based Blood Bank Management System. The goal
is to connect blood donation seekers (users / patients) with registered blood
banks across India.

Key capabilities:
- Users can search for blood banks by blood type, state, district, location.
- Users can place blood requests (orders) from a specific blood bank.
- Blood Banks can manage their blood stock inventory (add, increase, decrease).
- Blood Banks can accept or reject incoming blood requests.
- When a request is accepted, blood stock is automatically decremented.
- Admins have a dashboard with platform-wide statistics and oversight.
- Three distinct login portals: User, Blood Bank, and Admin.
- JWT-based authentication for all three roles.

4. DATABASE DESIGN (TABLES & RELATIONSHIPS)
================================================================================

The database is MySQL. All queries are written as raw SQL strings.

TABLES:
-------

1. users
   - id (PK, auto-increment)
   - full_name
   - email
   - phone (used as login credential)
   - password (plain text — no hashing currently)
   - state
   - district
   - address
   - user_type (default: 'person')
   - createdAt, updatedAt

2. blood_banks
   - id (PK, auto-increment)
   - name
   - email (used as login credential)
   - phone
   - password (plain text — no hashing currently)
   - state
   - district
   - address_lines
   - latitude, longitude (for geo-search)
   - availability_status (boolean, default: true)
   - operating_hours
   - verified (boolean, default: false)
   - createdAt, updatedAt

3. blood_stock
   - id (PK)
   - blood_bank_id_fk (FK → blood_banks.id)
   - blood_type (e.g., 'A+', 'B-', 'O+', 'AB+', etc.)
   - quantity (units available)
   - price_per_unit
   - createdAt, updatedAt
   - UNIQUE constraint: (blood_bank_id_fk, blood_type) — one row per blood type per bank

4. orders
   - id (PK, auto-increment)
   - user_id_fk (FK → users.id)
   - blood_bank_id_fk (FK → blood_banks.id)
   - blood_type
   - quantity
   - price_at_purchase (snapshot of price when order was placed)
   - status (ENUM: 'PENDING', 'COMPLETED', 'REJECTED')
   - urgency_level (ENUM: 'NORMAL', 'URGENT')
   - delivery_method (default: 'PICKUP')
   - createdAt, updatedAt

5. admins
   - id (PK)
   - username
   - password
   - role (default: 'admin')
   - createdAt, updatedAt

6. additions (stock addition record/audit log)
   - blood_type
   - quantity
   - blood_bank_id_fk

KEY RELATIONSHIPS:
------------------
  users        ──(1:N)──> orders
  blood_banks  ──(1:N)──> orders
  blood_banks  ──(1:N)──> blood_stock
  blood_stock.blood_type is unique per blood_bank (composite unique key)

5.1 ENTRY POINT & SERVER SETUP
-------------------------------
File: api/app.js
- Creates Express app
- Applies middleware: morgan logging, JSON body parsing, URL-encoded parsing,
  cookie-parser, CORS (open to all origins)
- Mounts all routes under /api/v1 via apiRouter
- Has a health check GET / endpoint that returns a success message
- Has a catch-all 404 handler and a global error handler (returns JSON)

File: api/bin/www (start script)
- Standard Express www server script, starts HTTP server on configured port

File: api/asyncHandler.js
- Wraps async route handler functions to auto-catch errors and pass them to
  Express's next() error handler — avoids try/catch in every controller.

File: api/utils/responses/ApiResponse.js — sendResponse(res, status, success, msg, data)
- Standardized response shape: { success, message, data }

File: api/utils/responses/successCodes.js
- Exports SUCCESS = true, FAILURE = false constants for consistent response flags


5.2 AUTHENTICATION SYSTEM
--------------------------
File: api/controllers/authController.js

Three separate login functions:
  a) adminLogin     — lookup by username in admins table
  b) userLogin      — lookup by phone in users table
  c) bloodBankLogin — lookup by email in blood_banks table

All three:
  - Validate input fields
  - Query DB for the entity
  - Compare password (plain-text comparison — NO bcrypt hashing currently)
  - On success: generate JWT (expires in 2h) signed with JWT_SECRET_KEY from .env
  - JWT payload includes: id, role, and the login identifier (username/phone/email)
  - Return user/admin/bloodBank object with token (password stripped out)

NOTE: Passwords are stored and compared in plain text. This is a security
      vulnerability that should be fixed before any production deployment
      (use bcrypt or argon2 for password hashing).


5.3 MIDDLEWARE
--------------
File: api/middlewares/authMiddleware.js

Exports an authMiddleware object with these guards:
  - isAdmin           — only allows role === 'admin'
  - isUser            — only allows role === 'user'
  - isBloodBank       — only allows role === 'bloodBank'
  - isAdminOrBloodBank — admin OR blood bank
  - isUserOrBloodBank  — user OR blood bank
  - isAdminOrUser      — admin OR user
  - isAuthenticated    — any authenticated role

All middleware uses verifyToken() which:
  - Extracts Bearer token from Authorization header
  - Verifies it against JWT_SECRET_KEY
  - Attaches decoded payload to req.user
  - Calls next() if valid, else 401/403 JSON response

  - getOrdersByBloodBank()   — orders for a specific blood bank
  - getOrdersByUser()        — orders for a specific user

  IMPORTANT — updateStatus():
    When updating to COMPLETED:
      1. Fetch the existing order details (blood type, quantity)
      2. Check current blood stock for that blood type at the relevant bank
      3. If insufficient stock → return 400 error with available count
      4. Update order status to COMPLETED
      5. Decrease blood stock by order quantity via bloodStocksModel.decreaseStock()
    When updating to REJECTED:
      - Just update order status (no stock interaction)

7. USER ROLES & PERMISSIONS
================================================================================

The system has 3 distinct roles, each with separate login flows:

ROLE 1: USER (regular blood seeker / patient)
  - Login via: phone + password
  - JWT role value: "user"
  - Can:
    - Browse blood banks (search by blood type, location)
    - Place blood requests (create orders)
    - View their own order history and status
  - Cannot: access admin or blood bank dashboards

ROLE 2: BLOOD BANK (registered blood bank institution)
  - Login via: email + password
  - JWT role value: "bloodBank"
  - Can:
    - View and manage their blood stock inventory
    - View incoming blood requests targeted at their bank
    - Accept (COMPLETED) or Reject blood requests
    - View their own stats (revenue, total orders, etc.)
  - Cannot: access admin dashboard, view other banks' data

ROLE 3: ADMIN (platform administrator)
  - Login via: username + password
  - JWT role value: "admin"
  - Can:
    - View platform-wide statistics and dashboards
    - List and manage all users
    - List and manage all blood banks
    - View all orders across the platform
    - Register new blood banks
  - Cannot: (all access in practice)

MIDDLEWARE COMBINATIONS AVAILABLE:
  - isAdmin, isUser, isBloodBank
  - isAdminOrBloodBank (for operations both can do)
  - isAdminOrUser (for user-related operations admin can also do)
  - isUserOrBloodBank
  - isAuthenticated (any logged-in role)

8. CORE BUSINESS LOGIC FLOW
================================================================================

FLOW 1: USER PLACES A BLOOD REQUEST
  1. User browses/searches blood banks for their blood type + location
  2. User selects a blood bank and blood type
  3. User submits request (blood type, quantity, urgency, delivery method)
  4. POST /api/v1/orders → ordersController.add()
  5. Order saved with status = 'PENDING'
  6. Blood bank receives the request in their dashboard

FLOW 2: BLOOD BANK ACCEPTS A REQUEST
  1. Blood bank views pending requests at GET /api/v1/orders/blood-bank/:id
  2. Blood bank opens request details
  3. Blood bank clicks "Accept"
  4. PUT /api/v1/orders/status → ordersController.updateStatus({ status: 'COMPLETED', ... })
  5. Backend fetches existing order (quantity, blood_type)
  6. Backend checks current stock: bloodStocksModel.getBloodStockByBloodType()
  7. If stock < requested quantity → return 400 error (cannot accept)
  8. If enough stock:
     a. Update order status to 'COMPLETED'
     b. Decrease blood stock by order quantity via bloodStocksModel.decreaseStock()
  9. Frontend shows success and updates order status

FLOW 3: BLOOD BANK REJECTS A REQUEST
  1. Same as above but status = 'REJECTED'
  2. No stock check needed
  3. Order status updated to 'REJECTED'

FLOW 4: BLOOD BANK ADDS BLOOD STOCK
  1. Blood bank navigates to stock management page
  2. Enters blood type, quantity (and optionally price)
  3. POST or PUT to blood stock endpoint
  4. bloodStocksModel.increaseStock() uses ON DUPLICATE KEY UPDATE
     to either insert new or add to existing quantity
  5. Stock count updated on dashboard

FLOW 5: SEARCH FOR BLOOD BANKS
  1. User submits search form (blood type, state, district, etc.)
  2. POST /api/v1/search with filter payload
  3. bloodBanksModel.search(filters) runs dynamic SQL with multiple optional filters
  4. Returns list of blood banks with:
     - available blood types (JSON array)
     - blood types + prices (JSON array of objects)
     - total quantity
  5. Frontend renders matching blood banks for user to choose from

10. KEY DESIGN DECISIONS & NOTES
================================================================================

1. NO PASSWORD HASHING:
   Passwords are stored as plain text in the database and compared as strings.
   This is a major security issue. Before production, ALL passwords should be
   hashed using bcrypt (or similar) at registration and compared at login.

2. NO ORM:
   Despite having Sequelize as a dependency, the project uses raw SQL queries
   via mysql2. This gives full SQL control but requires manual query management.

3. STANDARDIZED API RESPONSES:
   All API responses use sendResponse() for consistent shape:
   { success: true/false, message: "...", data: { ... } }

4. STOCK SAFETY:
   decreaseStock() uses SQL's GREATEST(quantity - ?, 0) to prevent negative stock.
   Additionally, order acceptance validates sufficient stock before decrement.

5. SOFT DELETE FOR BLOOD BANKS:
   Blood bank deletion sets availability_status = false rather than removing
   the record, preserving historical order data.

6. GEOLOCATION SUPPORT:
   Blood banks store latitude/longitude. The getNearbyBanks() function uses
   the Haversine formula directly in SQL to find banks within a given radius.

7. AUDIT TRAIL FOR STOCK ADDITIONS:
   addToAddtionRecord() logs every stock addition to an 'additions' table for
   tracking purposes.

8. JWT TOKEN STRUCTURE:
   - Admin token: { id: username, role: 'admin', username }
   - User token: { id: user.id, role: 'user', phone }
   - Blood Bank token: { id: bloodBank.username, role: 'bloodBank', email }
   All tokens expire in 2 hours.

9. CORS:
   CORS is currently configured with app.use(cors()) which allows ALL origins.
   This should be restricted to specific domains in production.

10. PRICE SNAPSHOT:
    Orders store price_at_purchase as a snapshot of the price at the time of
    ordering. This ensures historical data is not affected by future price changes.

11. DELIVERY METHOD:
    Orders support a delivery_method field (default: 'PICKUP'). Future versions
    may expand this to home delivery with address tracking.

12. URGENCY LEVELS:
    Orders have urgency_level (NORMAL or URGENT). The admin/blood bank dashboards
    separately count urgent pending orders for prioritization.
